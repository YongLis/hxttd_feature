# hxttd_feature 统一特征平台

## 背景与定位

在支付风控场景中，风险决策依赖海量特征——用户画像、设备指纹、交易行为、历史统计、第三方名单等。这些特征来源异构、计算逻辑复杂、时效性要求差异巨大（毫秒级在线决策 →
天级离线报表）。

hxttd_feature 定位于**支付风控统一特征平台**，提供从特征定义、计算、管理到服务的全链路能力。基于 Maven 多模块架构，支持
Spring Boot 微服务部署与 Flink 实时流式计算。

**技术选型：** Spring Boot 3.5、Apache Flink 2.2、Kafka、Redis (Redisson)、Nacos、Groovy/Aviator/Jexl 多语言脚本引擎、SPI
插件化连接器框架。

## 一期：特征计算引擎

### 设计理念

风控工程师的核心工作流是：**定义因子 → 配置依赖 → 编写表达式 → 发布上线**
。一期目标是将这条链路平台化、可配置化、可观测化，让风控策略迭代从"开发排期"变为"配置变更"。

### 核心能力

#### 1. 三层因子模型

```
                    ┌──────────────────┐
                    │   FeatureFactor  │  ← 特征因子（流速/聚合）
                    │  (滑动窗口聚合)     │
                    └────────┬─────────┘
                             │ 依赖
                    ┌────────▼─────────┐
                    │ DerivativeFactor │  ← 衍生因子（表达式+连接器）
                    │  (脚本+连接查询)    │
                    └────────┬─────────┘
                             │ 依赖
                    ┌────────▼─────────┐
                    │   MetaFactor     │  ← 元数据因子（字段直读）
                    │  (原始字段映射)     │
                    └──────────────────┘
```

- **MetaFactor**：映射 Raw Data 中的原始字段（如 `userId`、`amount`），由接入层透传
- **DerivativeFactor**：通过**条件脚本** + **数据源连接器** + **计算脚本**派生新因子，支持依赖 Meta/Feature/其他
  Derivative 因子，拓扑排序后层级并行计算
- **FeatureFactor**：基于 Flink 滑动窗口聚合的统计特征（如"近 24h 交易次数"、"近 7d 平均金额"）

#### 2. 多语言脚本引擎 (ttd-language)

| 引擎          | 定位         | 适用场景                 |
|-------------|------------|----------------------|
| **Groovy**  | 完整脚本语言     | 复杂条件判断、多步骤逻辑、自定义函数扩展 |
| **Aviator** | 高性能表达式     | 数值计算、简单布尔逻辑          |
| **Jexl**    | 轻量表达式（带缓存） | 高频简单表达式              |

三种引擎通过 `LanguageExecuteFactory` 统一调度，调用方只需指定 `ScriptType`
，无需感知底层实现。支持通过 `@FunctionDef` / `@MethodName` 注解扩展自定义函数并自动生成前端函数提示。

#### 3. SPI 插件化连接器框架

```
FeatureEngineService
        │
        ▼
FactorCalculateService
        │
   ┌────┴────┐
   ▼         ▼
Script     ConnectorService
           │
           ▼
      ConnectorEngine
           │
    ┌──────┼──────┬──────┐
    ▼      ▼      ▼      ▼
  JDBC    ES    HBase   HTTP
```

- 统一 `Connector` SPI 接口 + `AbstractConnector` 模板
- 拦截器链 (`ConnectorInterceptor`) + 观察者 (`ConnectorObserver`) 机制
- 内置 JDBC、Elasticsearch、HBase、HTTP 四种数据源插件
- 支持 `ConnectorFilter` 前置条件过滤与执行拦截

#### 4. 实时流速计算 (Flink)

5 个 Flink DataStream 任务，基于 Kafka Source + ValueState 状态管理：

| 任务    | 聚合函数 | 典型场景       |
|-------|------|------------|
| Count | 计数   | 近 1h 交易笔数  |
| Sum   | 求和   | 近 24h 交易总额 |
| Avg   | 均值   | 近 7d 平均客单价 |
| Max   | 最大值  | 历史最大交易金额   |
| Min   | 最小值  | 历史最小交易间隔   |

#### 5. 配置化管理后台

基于 Spring Boot + UmiJS + Ant Design Pro 的管理控制台，提供特征配置可视化、因子依赖拓扑管理、连接器配置、字典码表、审计日志等功能。14
个 Service 模块覆盖完整 CRUD + 变更追溯。

#### 6. 降级与容错

- `FallBackExecutor`：超时降级，支持自定义超时时间、默认值、异常值
- `ThreadPoolExecutor` + dynamic-tp 动态线程池
- `computingSet` 循环依赖检测（图算法在启动时拓扑排序 + 运行时守卫）
- Null 结果兜底：`ValueConvertor` 按类型返回安全默认值

## 二期：数据管道 & 特征平台

一期解决了"如何计算特征"。二期的核心命题是：**特征从哪里来、到哪里去、如何复用**。

### 目标架构

```
                         ┌──────────────────────────────┐
                         │     特征平台 (Feature Store)    │
                         │                              │
  ┌──────────┐           │  ┌──────────┐ ┌───────────┐  │
  │ 在线业务   │◄─────────┤──│ 在线特征   │ │ 特征管理后台 │  │
  │ (支付网关) │  ms级查询  │  │ (Redis)  │ │ (元数据)    │  │
  └──────────┘           │  └──────────┘ └───────────┘  │
                         │       ▲            ▲          │
                         │       │ 实时写入    │ 元数据同步 │
  ┌──────────┐           │       │            │          │
  │ 离线训练   │◄─────────┤──│ 离线特征   │            │  │
  │ (模型平台) │  批量获取  │  │ (Hive/OS) │            │  │
  └──────────┘           │  └──────────┘            │  │
                         │       ▲                     │
                         └───────┼─────────────────────┘
                                 │
                    ┌────────────┴────────────┐
                    │      数据管道 (二期)      │
                    │                          │
                    │  ┌────────────────────┐  │
                    │  │   Apache InLong    │  │  ← 数据集成与同步
                    │  │ (多源采集/ETL/分发)  │  │
                    │  └────────┬───────────┘  │
                    │           │               │
                    │  ┌────────▼───────────┐  │
                    │  │      Feast         │  │  ← 特征注册与服务
                    │  │ (特征定义/存储/服务)  │  │
                    │  └────────────────────┘  │
                    └──────────────────────────┘
```

### 2.1 Apache InLong — 数据集成层

一期中数据来源主要是支付网关实时请求 + Kafka 埋点。二期引入 InLong 构建**统一数据接入管道**：

| 能力         | 说明                                                      |
|------------|---------------------------------------------------------|
| **多源采集**   | 对接 MySQL Binlog、Kafka、HTTP、文件等多种数据源，统一采集格式              |
| **轻量 ETL** | 字段映射、格式转换、数据脱敏，入湖前标准化                                   |
| **多目标分发**  | 一路数据同时写入 Kafka（实时特征）、Hive/OSS（离线特征）、ClickHouse（OLAP 分析） |
| **运维管控**   | 可视化任务管理、指标监控、断点续传                                       |

**典型场景**：业务数据库的订单表变更 → InLong 采集 Binlog → 分发至 Kafka → Flink 实时聚合 → 在线特征 Redis。

### 2.2 Feast — 特征注册与服务层

一期中特征以 Java 对象配置存在，缺乏跨语言、跨场景的标准化描述。二期引入 [Feast](https://feast.dev/)（开源 Feature Store）实现：

| 能力                    | 说明                                                  |
|-----------------------|-----------------------------------------------------|
| **特征注册**              | 通过 Feast SDK 将特征元数据（名称、类型、来源、时效性）注册到 Feast Registry |
| **在线服务**              | Feast Serving 提供低延迟特征查询 API，替代自行维护的 Redis 查询逻辑      |
| **离线检索**              | Feast Offline Store 支持从数据湖批量拉取历史特征，统一训练数据生成         |
| **Point-in-Time 正确性** | 避免特征穿越问题：离线批量构建训练集时，按时间点精确关联特征快照                    |

**集成路线**：

```
一期现状:   配置文件 → FeatureConfiguration → 因子计算 → TxnParamContext

二期目标:   Feast Registry ← 特征元数据同步
                │
           Feast Serving ← 在线特征查询 (替代直连 Redis)
                │
           Feast Offline ← 离线特征批量拉取 (替代手动 Hive 取数)
```

### 2.3 特征平台统一服务

在 InLong + Feast 之上，构建统一的特征服务层：

- **特征市场**：可搜索、可浏览的特征目录，标注所有者、SLA、血缘关系
- **特征血缘**：从数据源 → InLong 采集 → Flink 聚合 → Feast 注册 → 模型消费，全链路可追溯
- **多语言 SDK**：Java / Python / Go SDK 统一接入，在线离线一致性保障
- **特征质量监控**：漂移检测、分布变化告警、时效性 SLA 看板

### 2.4 落地路径

| 阶段           | 内容                                                             | 交付物                |
|--------------|----------------------------------------------------------------|--------------------|
| **Phase 2a** | InLong 接入已有 Kafka 数据源，打通 MySQL Binlog → Kafka → Flink 链路       | InLong 集群 + 数据同步任务 |
| **Phase 2b** | Feast 集成，将一期 MetaFactor/DerivativeFactor 元数据同步至 Feast Registry | Feast 部署 + 元数据同步脚本 |
| **Phase 2c** | Feast Serving 替代直连 Redis，统一在线特征查询入口                            | 在线查询 SDK + 性能压测报告  |
| **Phase 2d** | Feast Offline 构建训练数据集管线，打通模型平台 → 特征平台链路的离线场景                   | 离线特征批量检索 API       |

## 总结

| 维度       | 一期（已交付）                                                  | 二期（规划）                           |
|----------|----------------------------------------------------------|----------------------------------|
| **数据接入** | 支付网关请求 + Kafka 埋点                                        | InLong 多源采集 + 多目标分发              |
| **特征定义** | Java 配置模型（MetaFactor / DerivativeFactor / FeatureFactor） | Feast FeatureView + Registry 标准化 |
| **实时计算** | Flink DataStream 5 类聚合                                   | 扩展至更多聚合类型 + 窗口策略                 |
| **在线服务** | 应用内 `TxnParamContext` + 直接 Redis 读写                      | Feast Serving 统一 gRPC/REST 查询    |
| **离线获取** | —                                                        | Feast Offline Store 批量检索         |
| **特征复用** | 项目内因子依赖                                                  | 跨项目特征市场 + 血缘                     |
| **质量保障** | Null 兜底 + 超时降级                                           | 漂移检测 + 分布监控 + SLA 看板             |

一期实现了特征计算的**引擎化**，二期将在之上构建特征管理的**平台化**——从"能算"到"能用、能管、能复用"。
